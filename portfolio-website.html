Portfolio Website

A static engineering portfolio built as a digital briefing package: the homepage gives a fast identity signal, while the project pages provide deeper evidence of architecture, implementation, verification, and documentation.

Static Website GitHub Pages HTML / CSS JavaScript Technical Writing Cloudflare Analytics

Overview

This portfolio exists because a resume, LinkedIn post, or short verbal explanation cannot fully show the process, difficulty, tradeoffs, or systems thinking behind technical work. A resume can say what was built, but it usually cannot show how the work was structured, why certain decisions were made, or what the builder's standard of technical communication looks like.

The site is designed as a communication system for multiple audiences. Recruiters need a quick signal. Hiring managers need evidence. Engineers need enough structure to see whether the thinking is real. The homepage gives the fast overview, and the project pages act as technical appendices.

The problem was not a lack of projects. The problem was that complex engineering work gets flattened when it is forced into a resume bullet, short post, or quick explanation. That format can miss the architecture, tradeoffs, verification work, and communication discipline behind the final result.

System Architecture

At the system level, this is a static website. The browser loads finished HTML, CSS, JavaScript, and assets rather than talking to a custom backend or database. That architecture fits the goal: documentation, presentation, and project evidence without unnecessary server-side complexity.

My Role

My role was to define the technical direction, decide what the portfolio needed to communicate, provide project context, review the design, correct inaccuracies, and control the final quality. AI tools supported iteration, but the project direction, project information, technical judgment, and approval decisions stayed with me.

Ownership Model

The site explicitly treats AI as a tool, not as the owner of the work. The important project decisions came from human review: what to include, what to remove, what to correct, and how to keep the technical story accurate.

Implementation

The implementation was organized around layered communication. The homepage introduces the identity and project set, while the project pages use a repeatable tab structure so each system can be explained in the same rhythm: overview, problem, architecture, tools, role, implementation, verification, and lessons learned.

Testing / Verification

Verification focused on whether the site communicates clearly, navigates correctly, works on different screen sizes, and presents project evidence without overstating the work.

Navigation

Checked homepage anchors, project-card links, back links, resume access, and LinkedIn access.

Responsive layout

Tested desktop and phone layouts, then adjusted mobile header behavior, tab wrapping, text scaling, and section spacing.

Interaction state

Verified tab switching and active navigation behavior so the site feels guided instead of static.

Accessibility semantics

Used tablist, tab, tabpanel, and aria-selected patterns to keep interaction state more understandable to assistive technologies.

Content accuracy

Reviewed project language to avoid overstating work and to keep technical descriptions aligned with the actual systems.

Deployment

Confirmed the repository structure works with GitHub Pages and that analytics data reaches Cloudflare after launch.

What I Learned

This project reinforced that a portfolio is not only a design task. It is a technical communication problem. The site has to help a viewer understand what was built, why it matters, how it was structured, and whether the builder can communicate work in a way that survives review.
